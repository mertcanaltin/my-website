*Mert Can Altin and [Daniel Lemire](https://lemire.me) (co-author, [simdjson](https://simdjson.org))*

## How it started

I was building a JSON Schema validator as a side project. The idea was simple: what if we used simdjson for parsing and RE2 for regex, instead of the standard JavaScript approach?

I shared the early results with Daniel Lemire, the creator of simdjson. He liked the concept and started contributing. His first PR optimized the bitmask operations in the C++ layer. Small change, big impact.

Then Matteo Collina from Fastify started opening issues on our repos. He didn't just give opinions, he dug into the code and pointed out real problems.

His first issue on ata-validator was about our benchmarks. We were comparing ata's C++ JSON parsing against ajv's `JSON.parse()`, which wasn't fair because Fastify already handles JSON parsing. In a real Fastify app, what matters is `validate(req.body)` performance, not JSON string parsing. He was right. We rewrote the benchmarks to measure what actually matters.

He opened another issue on fastify-ata asking about default values. ajv applies defaults to objects during validation, and ata didn't support that. We added it.

He also caught us claiming "CSP safe" in our docs, which doesn't make sense for a Node.js native addon. We removed it.

Each issue pushed us to be more honest and more complete. He then pointed us to Manuel Spigolon's [blog post](https://backend.cafe/how-to-unlock-the-fastest-fastify-server-startup) about Fastify startup optimization and said we should benchmark against that approach. That's when everything changed.

## The cold start problem

ajv has been the standard JSON Schema validator for 12 years. Every Fastify, Express, and OpenAPI tool depends on it. But ajv has a structural limitation: when you call `compile(schema)`, it compiles everything immediately. In a Fastify app with 100 routes, all 100 schemas compile at startup, even if only 3 will receive requests that day.

Manuel Spigolon wrote a [blog post](https://backend.cafe/how-to-unlock-the-fastest-fastify-server-startup) about this problem. His solution was a build/read pattern: pre-compile schemas to disk, load them at startup. It works, but requires extra tooling and configuration.

We took a different approach.

## Lazy compilation

In ata-validator, the constructor does nothing. It stores the schema and sets up stub functions. When the first `validate()` call comes in, only that schema compiles. The rest stay dormant until needed.

```javascript
const v = new Validator(schema); // ~0.5 microseconds, no compilation
v.validate(data);                // compiles here, on first use
v.validate(data);                // already compiled, zero overhead
```

We also added a schema cache. If the same schema string appears again (common in test suites and microservices), the compiled function is reused. No `new Function()` overhead.

The result: 761x faster first validation than ajv, measured with mitata process isolation.

## Profiling-driven optimization

Daniel taught me something valuable: profile first, optimize second. Don't guess where the slowness is. Measure each part, find the bottleneck, fix it, measure again.

We applied this systematically to the JS codegen:

**Finding 1: `new Set()` was the bottleneck.**

For `additionalProperties: false`, we were creating a Set of allowed keys and looping through object keys on every validation call. Replacing this with a simple property count check (`Object.getOwnPropertyNames(d).length !== 4`) gave us a 10x improvement on that code path.

**Finding 2: Regex was recreated per call.**

Our codegen put `new RegExp('^[0-9]{5}$')` inside the generated function body. Every `validate()` call compiled the regex from scratch. Moving it to a closure (compiled once, reused via factory pattern) gave us 1.8x.

**Finding 3: Destructuring was slower than direct access.**

We thought V8 would optimize `const {"id": _h0} = d` (destructuring batch read) better than `d["id"]` (direct access). Profiling showed the opposite. Direct property access with V8's inline cache was faster. Simpler code, fewer local variables, better JIT.

**Finding 4: Required checks were redundant.**

We were checking `if (d.id === undefined) return false` and then later `if (!Number.isInteger(d.id)) return false`. But `Number.isInteger(undefined)` already returns `false`. The type check catches missing properties for free. Removing the redundant required check eliminated one property access per required field.

**Finding 5: additionalProperties check placement matters.**

Moving the property count check from the beginning to the end of the function let V8 optimize the type checks first. Valid data (the common case) runs through fast type checks before hitting the more expensive property enumeration.

Each optimization was small. Together they took isValidObject from 13M to 215M ops/sec.

## The typebox comparison

[@Revenode](https://x.com/Revenode) on X asked "How fast is it compared to TypeBox?" We benchmarked it honestly.

On a simple schema without format validators (typebox 0.x didn't support them), the numbers were close. Then sinclairzx81, TypeBox's creator, pointed out that TypeBox 1.x is fully JSON Schema compliant with RFC format support.

We updated the benchmark to use the same raw JSON Schema on both sides, with `format: 'email'` included. The results:

- Boolean check: ata 4.9x faster
- Compile: ata 108x faster
- First validation: ata 80x faster

The format validator difference is because ata uses hand-written parsers (indexOf-based email check) while most libraries use regex. No regex means no backtracking overhead.

## What we learned

**Profile before optimizing.** We wasted time guessing at bottlenecks early on. The profiling approach (add one keyword at a time, measure the drop) found problems we never would have guessed.

**Lazy beats eager.** Not compiling things you don't need sounds obvious, but ajv's architecture makes it impossible. We had the advantage of designing from scratch.

**Be honest about numbers.** Matteo caught us with misleading benchmarks early on (comparing JSON string validation without accounting for JSON.parse). We fixed it and learned to always show what's being measured, under what conditions. Every benchmark file in the repo has notes explaining what it tests and what it doesn't.

**Listen to feedback.** Every major improvement came from someone else's suggestion. Matteo pointed us to cold start. Lemire taught us profiling. [@sinclairzx81](https://x.com/sinclair_zx81) told us about typebox 1.x format support. [@Revenode](https://x.com/Revenode) suggested mitata for process isolation.

## Current state

ata-validator is open source, published on npm, and used with Fastify via the fastify-ata plugin. Daniel Lemire is a co-author. We have 98.4% spec compliance on the JSON Schema Test Suite (Draft 2020-12).

The numbers, measured with mitata:

- Schema compilation: 2,083x faster than ajv
- First validation: 761x faster than ajv
- isValidObject: 3.9x faster than ajv
- validate(obj) valid: 1.9x faster than ajv
- vs typebox 1.x: 4.9x faster (with format validators)
- ReDoS protection: 2,391x faster (RE2 regex)

npm install ata-validator
